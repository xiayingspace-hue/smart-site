## 4. 转换规则详述

### 4.1 Epic 划分规则

**步骤**:

1. **从主流程切分**:把需求 §6.1 主流程按"业务里程碑"切成几段,每段一个 Epic
2. **横切 Epic 识别**:有些 Epic 不在主流程上(如归档、审计、管理后台),从其他章节识别
3. **Epic 命名**:动词 + 名词,例如「资源上传与发布」、「审批与流转」
4. **粒度判断**:一个 Epic 通常包含 3~10 个 Story;超过则拆分,不足则合并

**输出格式**:

```markdown
| Epic ID | 名称 | 描述 | 包含 Story | 优先级 |
|--------|------|------|-----------|-------|
| EPIC-01 | [Epic 名] | [一句话描述] | US-001 ~ US-005 | P0 |
```

---

### 4.2 Story 拆分规则(核心)

**核心拆分维度**(优先级从高到低):

| 维度 | 适用场景 | 示例(抽象) |
|-----|---------|------------|
| **角色** | 同一功能不同角色行为不同 | 角色 A 操作是一个 Story,角色 B 处理是另一个 |
| **流程节点** | 一个流程多步骤 | 步骤 1 一个 Story,步骤 2 一个 Story |
| **正向 vs 异常** | 异常路径独立 | 主路径一个,断网恢复一个 |
| **CRUD 操作** | 资源型功能 | 创建/列表/详情/删除分开 |
| **批量 vs 单个** | 复杂度差距大 | 单个操作一个 Story,批量操作另一个 |

**INVEST 检查**(每个 Story 必须满足):

- [ ] **Independent**: 不强依赖其他 Story 完成才能验收
- [ ] **Negotiable**: 描述抽象到价值层面,不是技术指令
- [ ] **Valuable**: 单独交付有价值
- [ ] **Estimable**: 工作量估得出来
- [ ] **Small**: 1 个 Sprint 内可完成
- [ ] **Testable**: 关联 AC 明确

**Story 必填字段**:

| 字段 | 必填 | 来源 |
|-----|-----|------|
| 故事描述(角色/想要/以便) | ✅ | 标准格式 |
| Epic | ✅ | §1 |
| 优先级 | ✅ | 见 §4.5 派生规则 |
| 估时 | ✅ | 见下方估时表 |
| 前置依赖 | ✅ | 见 §4.3 |
| 所属流程节点 | ✅ | 引用需求 §6 章节 |
| 关联 AC | ✅ | 列出所有 AC ID |
| 验收要点 | ✅ | 简述,详细见 requirement.md §8 |
| 涉及接口 | 🔶 | 若 data-contract 已生成,引用 API ID |
| 技术风险 | 🔶 | 识别明显风险点 |
| 已知问题 | 🔶 | 引用 OQ-XXX |

**估时规则**:

| 复杂度 | 估时(总人日) | 判定特征 |
|-------|-------------|---------|
| XS | 1-2 | 仅前端,无新接口 |
| S | 3-5 | 前后端各有少量改动 |
| M | 5-10 | 完整 CRUD + UI |
| L | 10-20 | 涉及状态机、异步任务 |
| XL | > 20 | **必须再拆分**,不允许直接交付 |

---

### 4.3 依赖识别规则

**自动识别依赖的方法**:

1. **数据依赖**: Story B 操作的资源由 Story A 创建 → B 依赖 A
2. **流程依赖**: §6 主流程中 B 在 A 之后 → B 依赖 A
3. **基础设施依赖**: B 用到的接口由 A 实现 → B 依赖 A

**循环依赖检测**: 若 A 依赖 B,B 依赖 A,**报错**并提示需要进一步拆分。

---

### 4.4 Sprint 排布建议规则

**排布原则**:

1. **MVP 优先**: 第一个 Sprint 必须能交付一个端到端最小闭环
2. **依赖前置**: 被依赖的 Story 必须在前面 Sprint
3. **并行最大化**: 同 Sprint 内尽量是相互独立的 Story
4. **风险前置**: 技术风险高的 Story 早做

**单 Sprint 容量假设**: 默认 ~20 人日(可根据团队调整)。本 agent 不强行平衡,只给建议。

---

### 4.5 优先级派生规则

| 来源 | 推导 |
|-----|------|
| 需求文档 priority 字段 | 整个需求的基础优先级 |
| Story 关联 AC 中包含 P0 AC | Story 优先级 ≥ P1 |
| Story 是其他 Story 的前置依赖 | 优先级提升 |
| Story 在主流程上 | 优先级 ≥ P1 |
| Story 仅在异常或边缘场景 | 优先级 ≤ P2 |

冲突时取更高优先级。

## 6. 校验清单

通用清单外,本 agent 自检:

- [ ] 每个 Story 都引用至少 1 个 AC
- [ ] 每个 AC 至少被 1 个 Story 引用(否则该 AC 无人交付)
- [ ] 没有 XL 级 Story
- [ ] 没有循环依赖
- [ ] 每个 Story 满足 INVEST 自检
- [ ] Sprint 排布与依赖图一致
- [ ] 所有 Story 的角色都在需求 §2.1 定义过
- [ ] 至少识别出 1 个 Epic

---

## 7. 禁止事项

1. ❌ 把多个用户角色的行为塞进同一个 Story("用户操作"过于笼统)
2. ❌ 把"前端做 X" / "后端做 Y" 写成 Story(那是任务,不是 Story)
3. ❌ 自动估时为某个固定值(必须按复杂度规则)
4. ❌ 给 PM 没标的 AC 自动归类到 Story
5. ❌ 把 OQ 待定项当作已确认事项放进验收要点
6. ❌ 在 Story 里重新定义 AC(只引用 ID)

---

## 8. 拆分判断流程

```
对每个潜在 Story:
  1. 角色是否单一? 否 → 按角色拆分
  2. 涉及多个流程节点? 是 → 按节点拆分
  3. 估时是否 XL? 是 → 找拆分维度继续拆
  4. 异常路径是否复杂? 是 → 异常路径独立 Story
  5. 是否同时改前端 + 后端 + 异步任务? 是 → 考虑按层拆
  6. INVEST 是否全过? 否 → 调整
  7. 通过 → 记录为 Story
```
