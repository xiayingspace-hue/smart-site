# 术语表规则(Glossary Rules)

> **本文档定义 glossary-agent 的转换规则**:从 `requirement.md` 提取业务术语,生成或增量更新 `glossary.md`。
>
> 注意:glossary 通常是**项目级**的,跨多个需求共享。本 agent 默认行为是**增量更新**,不是每次重建。

继承:`global-rules.md`、`traceability-rules.md`

---

## 1. 角色定义

| 项 | 值 |
|---|---|
| Agent ID | glossary-agent |
| 输出文件 | `glossary.md` |
| 服务对象 | 全员(PM/UI/前端/后端/QA) |
| 职责边界 | 业务术语提取、命名规范、禁用词识别、同义词检测 |
| **不做** | 业务规则定义、字段命名(那是 data-contract 的事) |

---

## 2. 输入契约

### 2.1 必读输入

| 来源 | 章节 | 用途 |
|-----|------|------|
| `requirement.md` | 全文 | 提取所有业务术语 |
| 已有的 `glossary.md`(若存在) | 全文 | 增量合并,**不覆盖**既有定义 |
| `data-contract.md`(若存在) | §2 枚举 | 同步状态术语 |

### 2.2 增量更新模式

本 agent 的特殊行为:

- **首次生成**: 从需求全文扫描,初始化 glossary
- **再次运行**: 加载既有 glossary,**只新增**术语,**不修改**已有条目
- **修改既有条目**: 必须由人类操作,agent 不主动改写

---

## 3. 输出章节映射表

| 输出章节 | 输入来源 | 转换规则 |
|---------|---------|---------|
| §1 业务术语 | 需求全文名词扫描 | 见 §4.1 |
| §2 角色术语 | 需求 §2.1 | 直接抄录 |
| §3 状态术语 | 需求 §5.1 + data-contract §2.2 | 直接抄录,保持代码一致 |
| §4 技术术语缩写 | 模板预填项 + 需求中的英文缩写 | 见 §4.2 |
| §5 命名规范 | 模板固定内容 + 项目特殊约定 | 一般保持模板 |
| §6 禁用词与常见错误 | 需求中识别的同义词漂移 | 见 §4.3 |

---

## 4. 转换规则详述

### 4.1 业务术语提取规则

**步骤**:

1. **名词频率统计**:
   - 扫描需求全文,统计专有名词出现次数
   - 出现 ≥ 3 次的核心名词 → 进入 candidate 列表
   - 出现 1-2 次但属于业务核心(实体名、操作名)→ 进入 candidate

2. **过滤规则**(以下不进 glossary):
   - 通用词汇:用户、系统、功能、需求、页面、按钮、接口、数据
   - 技术术语:API、SQL、HTTP、JSON 等(进 §4 缩写表,不进 §1)
   - 形容词、副词:重要、快速、稳定

3. **生成条目格式**:

   ```markdown
   | 术语(中文) | 术语(英文/代码) | 定义 | 同义词/易混淆 | 反例 |
   |-----------|----------------|------|--------------|------|
   | [中文] | <!-- TODO: 英文待定 --> | [从需求上下文提取的定义] | <!-- TODO --> | <!-- TODO --> |
   ```

4. **定义提取规则**:
   - 若需求中有"[术语]:[定义]"的显式定义 → 直接使用
   - 否则从首次出现的上下文截取一句话作为定义
   - 定义长度控制在 30-80 字

5. **英文/代码命名**:
   - **不自动翻译**(易出错)
   - 标注 `<!-- TODO: 英文翻译 -->`,等人工确认
   - 例外:已有标准译法的(如"用户" → "user")可填,但加 `<!-- DERIVED -->`

6. **同义词识别**:
   - 扫描需求全文,寻找指代相同概念的不同名词
   - 输出到"同义词/易混淆"列,**并提示需要 PM 确认主称呼**
   - **不擅自判定**哪个对哪个错

---

### 4.2 技术缩写提取规则

**保留模板预填项**: AC / US / TC / OQ / API / DTO / SSoT(这些是文档体系自身的术语)

**新增规则**:
- 扫描需求中出现的英文大写缩写(2-5 字母)
- 上下文能识别全称的 → 添加
- 上下文识别不出的 → 标 TODO

---

### 4.3 禁用词识别规则

**典型同义词漂移场景**:

| 模式 | 处理 |
|-----|------|
| 同一段落两个不同名词指代同一概念 | 提示需要选其一作为标准 |
| 中英混用 | 中文作为标准,英文留 §1 第二列 |
| 缩写与全称并存 | 由 PM 选定主用法 |

**禁用词条目示例(模板,无业务内容)**:

```markdown
| ❌ 不要用 | ✅ 应该用 | 原因 |
|---------|---------|------|
| [缩写] | [全称] | 缩写易引起误解 |
```

> ⚠️ **agent 不主动判定哪个是错的**,只提示"以下两个词疑似同义",由 PM 决策。

---

## 5. 命名约定(本 agent 输出 §5)

通常保持模板默认。如项目有特殊约定,在此追加,但保持总体一致。

---

## 6. 校验清单

通用清单外,本 agent 自检:

- [ ] 每个术语条目 5 列都有内容(允许 TODO,但不能空)
- [ ] §1 中没有重复术语
- [ ] §3 状态术语与 data-contract.md §2.2 完全一致(若 data-contract 已生成)
- [ ] §2 角色术语与 requirement.md §2.1 完全一致
- [ ] 没有自动翻译的英文(都应是 TODO 或人工确认过的)

---

## 7. 增量更新校验(再次运行时)

```pseudo
function update_glossary(new_terms, existing_glossary):
  for term in new_terms:
    if term in existing_glossary:
      log("skipped: %s already defined" % term)
    else:
      append_to_glossary(term)
      log("added: %s" % term)

  for term in existing_glossary:
    similar = find_similar_in(new_terms, term)
    if similar:
      warn("potential synonym: %s vs %s" % (term, similar))
```

---

## 8. 禁止事项

1. ❌ 修改已有术语的定义(由人类决策)
2. ❌ 自动翻译中文为英文
3. ❌ 删除既有条目
4. ❌ 把数据契约里的字段名当业务术语放进 glossary
5. ❌ 把通用词(用户/系统)放进 §1
6. ❌ 自动判定同义词中"哪个对哪个错"

---

## 9. 与下游 agent 的协作

glossary 生成/更新后,下游所有 agent **必须**:

- 在生成时引用 glossary 的术语
- 发现自己想用的词不在 glossary 中 → 报告给 PM,**不自创**
- 不能用同义词替换 glossary 已定义的词
