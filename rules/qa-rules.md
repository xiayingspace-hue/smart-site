## 4. 转换规则详述

### 4.1 测试策略规则

**测试金字塔默认占比**:
- 单元 60% / 集成 25% / 契约 5% / E2E 10%

**调整因素**:
- 系统涉及复杂状态机 → 单元占比提高
- 微服务多 → 契约占比提高
- 用户路径关键 → E2E 占比提高

**分工规则**(关键):
- 单元 + 集成:开发负责
- 契约:QA 负责
- E2E:QA 负责

---

### 4.2 测试场景生成规则(核心)

#### §3.1 主流程场景

**输入**: requirement §6.1 主流程

**规则**:
- 每条主流程 → 1 个 SC-XXX
- 涉及多角色 → 按角色拆分多个场景
- 涉及多种正常分支 → 各派生一个场景

#### §3.2 异常场景

**输入**: requirement §6.2 异常流程

**规则**:
- 每个异常 → 1 个 SC-EXX
- 关联 AC:扫描 §8 AC 中 Given 或 When 描述了异常的

#### §3.3 权限场景

**输入**: requirement §3 权限矩阵

**规则**:
- 矩阵中每个 ❌ 单元格 → 派生 1 个越权测试 SC-PXX
- 至少包括:有权限正常通过 / 无权限被拒 / 越权访问被拒 / 资源不存在被拒(避免信息泄漏)

#### §3.4 状态转换场景(关键)

**输入**: data-contract §3 状态机

**规则**:
- **每条 transition** → 至少 1 个 SC-TXX(合法转换测试)
- **每条 forbidden_transitions** → 至少 1 个 SC-T-FXX(非法转换测试)
- guard 多条件 → 派生多个测试(条件全满足 / 单条件失败)

> ⚠️ **状态机覆盖率必须 100%**。本 agent 自检时必须校验。

---

### 4.3 测试用例(TC)生成规则(核心)

**为每个 AC 派生 TC**:

#### 4.3.1 必填 TC 数量(由 traceability §4.1 强制)

| AC 类型 | 最低 TC 数 | 派生维度 |
|--------|----------|---------|
| 主流程 AC | ≥ 2 | 1 正向 + 1 反向 |
| 边界值 AC | ≥ 4 | min, max, min-1, max+1 |
| 权限 AC | ≥ 2 | 有权限,无权限 |
| 状态转换 AC | ≥ 2 | 合法,非法 |
| 异常 AC | ≥ 1 | 触发异常路径 |

#### 4.3.2 TC 必填字段

```yaml
tc_id: TC-XXXX-XXX-NN          # 必填,全局唯一
covers_ac: [AC-XXXX-XXX]       # 必填,显式关联
scenario: SC-XXX               # 必填,关联场景
priority: P0 / P1 / P2         # 必填,继承 AC 优先级
test_level: unit / integration / contract / e2e  # 必填
preconditions: [...]           # 必填
test_data: [...]               # 必填(可为空数组)
steps:
  given: ...
  when: ...
  then: [...]
cleanup: [...]                 # 推荐
```

#### 4.3.3 测试层级分配规则

| AC 内容 | 推荐层级 |
|--------|---------|
| 工具函数 / 纯逻辑 | unit |
| 含 DB/缓存/MQ 的业务逻辑 | integration |
| 接口契约 | contract |
| 用户端到端流程 | e2e |
| 跨多服务流程 | e2e + contract |

#### 4.3.4 派生规则(伪代码)

```pseudo
for each AC in requirement §8:
  base_tc = {
    tc_id: generate_id(AC),
    covers_ac: [AC.id],
    given: AC.given,
    when: AC.when,
    then: AC.then
  }

  # 派生反向 TC(若主流程 AC)
  if AC.is_main_flow:
    add_tc(invert_when(AC))

  # 派生边界 TC(若涉及数值/长度约束)
  if AC.has_numeric_constraint:
    add_boundary_tcs(AC)

  # 派生权限 TC(若涉及角色)
  if AC.has_role_constraint:
    add_permission_tcs(AC)
```

#### 4.3.5 缺失处理

- AC 残缺(没有 Then)→ TC 标 `<!-- TODO: 期望结果待 PM 补充 -->`,**不擅自填**
- AC 含 OQ → TC 标 `<!-- TODO: 等待 OQ-XXX 解决 -->`

### 4.4 边界值与等价类规则

**输入**: data-contract.md §7 数据约束 + 各字段 max/min

#### 4.4.1 数值/长度边界

每个有数值或长度约束的字段必须派生 4 类用例:

| 等价类 | 边界 | 期望 |
|-------|------|-----|
| 有效 | 最小值 | 接受 |
| 有效 | 最大值 | 接受 |
| 无效 | 最小值-1 | 拒绝 |
| 无效 | 最大值+1 | 拒绝 |

#### 4.4.2 文本字段 fuzzing(标准集)

每个文本字段必须派生:
- 含 emoji
- 含中文/多语言
- 含特殊符号
- SQL 注入字符串
- XSS 字符串
- 超长文本(超出限制)
- 空字符串(若必填)
- 仅空格

#### 4.4.3 列表分页

每个列表 API 必须派生:
- page=0(无效)
- page=1, size=1(最小)
- page=1, size=max(最大)
- page=1, size=max+1(无效)
- 翻到不存在的页
- 排序字段非法

---

### 4.5 接口测试(契约测试)规则

**输入**: data-contract.md §4

#### 4.5.1 通用契约校验

强制套用模板内容,推荐用 schemathesis / pact 等工具基于 OpenAPI 自动生成。

#### 4.5.2 每个 API 的专项接口测试

每个 API 必须有以下 TC(标准模板):

| TC | 描述 | 期望 |
|----|------|-----|
| 01 | 正常请求 | 200 |
| 02 | 缺必填字段 | 400 |
| 03 | 字段类型错误 | 400 |
| 04 | 字段超长 | 400 |
| 05 | 未登录(若需鉴权) | 401 |
| 06 | 错误角色(若有权限) | 403 |
| 07 | 资源不存在(若涉及具体资源) | 404 |
| 08 | 业务冲突(若有唯一性) | 409 |
| 09 | 速率限制(若有限流) | 429 |
| 10 | 幂等性(若声明幂等) | 重复返回相同结果 |

> ⚠️ data-contract §4.2 错误码表中**每个错误码**都必须有对应 TC。否则报告"未覆盖错误码"。

---

### 4.6 性能测试规则

**输入**: requirement.md §9.1 + backend-spec.md §12

**规则**:

1. **每个明确性能目标的接口** → 1 个压测场景 PT-XXX
2. **场景类型**:
   - 稳态压测(目标 QPS,持续 10min)
   - 峰值压测(目标 QPS × 1.5)
   - 极端数据压测(单租户大数据量)
3. **通过条件**: 必须明确 P95 延迟阈值 + 错误率 < 阈值

**派生规则**:
- 列表接口 → 必加大数据量场景
- 写接口 → 必加并发场景
- 涉及大文件 → 必加文件大小边界场景

---

### 4.7 安全测试规则

**强制覆盖 OWASP Top 10**(模板已列):

至少覆盖 A01 越权、A03 注入、A07 鉴权、A09 日志监控四项。其他按业务相关性。

**派生规则**:
- 涉及文件上传 → 必加 magic number 校验、病毒文件、路径穿越测试
- 涉及外部下载 → 必加签名 URL 越权测试
- 涉及搜索 → 必加 SQL/NoSQL 注入测试
- 涉及富文本 → 必加 XSS 测试

---

### 4.8 测试数据规则

**fixtures 目录结构**(建议):

```
fixtures/
├── valid/        # 边界正常数据
├── invalid/      # 边界异常数据
└── security/     # 安全测试样本
```

**种子数据**: 每个角色至少 2 个用户、每个核心实体至少 1 条种子数据。

**数据生成**: 列出生成方式(SQL 脚本 / Faker / 手工 fixture),不擅自决定。

---

### 4.9 AC 覆盖矩阵规则(核心)

**输出强制结构**(traceability §3 + 本 agent 增强):

```markdown
| AC ID | 描述简述 | 覆盖的 TC | 状态 |
|------|---------|----------|------|
```

**规则**:

1. 列出 requirement.md **全部** AC,无一遗漏
2. 每个 AC 的"覆盖的 TC"列必须填具体 TC ID(逗号分隔)
3. 状态值:✅ DONE / 🟡 PARTIAL / ❌ TODO
4. 加自动校验脚本伪代码区块,提示团队启用 CI 校验

**双向校验**:
- 每个 AC → 至少 1 个 TC
- 每个 TC → 至少 1 个有效 AC

---

### 4.10 回归测试范围规则

**判定**: 哪些存量功能受本次需求影响?

**输入**:
- requirement §11 依赖与外部系统
- requirement.md `depends_on` / `blocks` front matter

**派生规则**:
- 复用了某模块的代码 → 该模块需回归
- 修改了共享数据模型 → 所有引用该模型的功能需回归
- 改了权限模型 → 所有受权限影响的功能需回归
- 加了新事件 → 所有订阅了相关事件的消费者需回归

## 7. 禁止事项

通用禁止外,本 agent 额外:

1. ❌ 重写 AC 描述(只在 TC 中引用 AC ID,不复制 AC 全文)
2. ❌ 自动给定性能阈值(必须从需求/backend-spec 来源)
3. ❌ 编造测试数据具体内容(列出"需要什么样的数据"而不是给具体值)
4. ❌ 生成可运行的测试代码(那是 test-code-agent 的事)
5. ❌ 为不存在的 AC 写 TC
6. ❌ 跳过状态机非法转换测试
7. ❌ 跳过 §11 中"未覆盖" AC 的标记(必须显式标 ❌ 而不是不列)
8. ❌ 把 OQ 待定项当作已确认事实写进 TC then
