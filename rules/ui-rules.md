# UI 设计说明生成规则(UI Rules)

> **本文件定义如何从需求文档生成 `ui-spec.md`**。
>
> UI agent 的工作不是"画图",是把需求中的业务规则**翻译成视觉/交互结构**,供设计师与设计 agent 据此产出 Figma。

## 3. 转换规则

### 3.1 信息架构推导(§2)

从 `requirement.md §6.1` 主流程识别页面:

| 流程关键词 | 推断为 |
|----------|------|
| "进入 X 页面" / "选择 X" | 一个页面 |
| "查看列表" | 列表页 |
| "查看详情" | 详情页 |
| "填写/编辑" | 表单页或浮层 |
| "选择/挑选" | 选择器(picker) |
| "确认" | 确认对话框 |

**导航识别**:
- 主入口:模块在产品全局导航中的位置(若需求未指定 → 在 OQ 中登记)
- 二级入口:从主页面到子页面的跳转
- 反向入口:通知/搜索结果/外部链接进入的目标页

### 3.2 页面/组件清单(§3)

#### 3.2.1 每个页面的输出结构(强制)

每个页面必须包含以下子节,缺一不可:

```markdown
### 3.X [页面名]

**关联 Story**: US-XXX
**关联 AC**: AC-XXXX-XXX

**布局**: <顶部/主体/侧栏/底部 各放什么>

**字段展示**:
<引用 data-contract §1 的字段,不重复定义类型>

**关键交互**: <表格>

**状态(必填 5 态)**: <表格,5 行>

**权限可见性**: <表格,角色 × UI 元素>
```

#### 3.2.2 5 态生成规则(强制)

**任何展示业务数据的页面,必须输出 5 种状态**。缺一即视为不合格。

| 状态 | 必填内容 |
|-----|---------|
| 空 | 触发条件 + 视觉表现 + 文案 + CTA(如有) |
| 加载 | 骨架屏或 spinner 选择 + 范围(全屏/局部) |
| 正常 | 默认展示样式 |
| 错误 | 接口失败 / 权限失败 / 网络断 三种细分 |
| 极端数据 | 超长文本截断、超大列表(> 1000)处理、超多选项处理 |

**降级**:页面不展示动态数据(纯静态)→ 标注"本页面无 5 态需求"。

#### 3.2.3 权限可见性矩阵(强制)

对每个页面/组件,根据 `requirement.md §3` 的权限矩阵,输出 UI 层"显示/隐藏/禁用":

| 推断规则 |
|---------|
| 角色对操作 = ✅ → 按钮 **显示** |
| 角色对操作 = ❌ → 按钮 **隐藏**(默认)或 **禁用**(若需告知用户该功能存在) |
| 角色对操作有条件(如"仅自己创建的") → 按钮显示但点击时弹提示 |

**默认策略**:**隐藏 > 禁用**。隐藏更安全,避免泄露功能信息。除非业务明确希望"提示用户升级权限"。

### 3.3 状态徽章映射(§4)

`data-contract.md §2` 中每个状态枚举,生成视觉 token:

| 状态语义 | 默认色彩 |
|---------|---------|
| 草稿 / 未开始 / 待办 | 灰(neutral) |
| 进行中 / 处理中 | 蓝(info) |
| 成功 / 已发布 / 已完成 | 绿(success) |
| 警告 / 待审核 / 已超期(可恢复) | 橙(warning) |
| 失败 / 错误 / 拒绝 | 红(danger) |
| 已归档 / 已停用(终态) | 深灰 |

**降级**:状态语义不明显时,使用 neutral,在 OQ 中登记。

### 3.4 异常流程 → 错误 UI

`requirement.md §6.2` 每个异常,在对应页面的"错误状态"中体现。

| 异常触发 | UI 表现 |
|---------|---------|
| 输入校验失败 | 字段下方红色文案,字段边框变红 |
| 接口失败(可重试) | 全局 toast + 重试按钮 |
| 权限失败 | 浮层式提示 + 返回 |
| 网络中断 | 顶部 banner,持续显示直到恢复 |
| 资源不存在 | 空状态页 + 返回链接 |
| 唯一性冲突 | 模态对话框 + 选项(覆盖/重命名/取消) |

### 3.5 复用判断(§8)

| 判定标准 | 决策 |
|---------|-----|
| 已有 DS 组件 + 满足业务校验 | 直接用 |
| 已有 DS 组件 + 需要业务包装 | 用 DS 组件 + 写业务包装组件 |
| 业务校验复杂(多种规则、状态依赖) | 新建业务组件 |
| 跨需求高频复用 | 新建,提议升级到 DS |
| 一次性、单页面使用 | 内联实现,不抽组件 |

### 3.6 文案规范(§9)

| 来源 | 处理 |
|-----|------|
| 按钮文案 | 动词开头("提交"而非"提交操作") |
| 错误文案 | 主谓宾完整,告诉用户**怎么办**而不仅是**出了什么事** |
| 空状态文案 | 友好 + CTA |
| 业务术语 | 严格使用 glossary.md §1 标准词,不允许漂移 |

**降级**:UI agent 不产出最终文案,只列文案位置和**临时文案**,标注"待文案/UX 写作 review"。

### 3.7 响应式策略

| 断点 | 默认处理 |
|-----|---------|
| Desktop(≥1280) | 完整布局 |
| Tablet(768~1280) | 侧栏折叠、表格转卡片或保持 |
| Mobile(<768) | 表格转卡片、操作浮动按钮、表单单列 |

requirement §9.4 若指明"不支持移动端" → 标注"本期仅 Desktop",移动端章节填"不适用"。

### 3.8 A11y 默认要求

- 颜色对比度 正文 ≥ 4.5:1,大字 ≥ 3:1
- 不仅靠颜色传递信息(状态徽章必须含文字或图标)
- 所有可交互元素 Tab 可达
- 焦点态明显
- 表单字段有 label
- 错误信息用 `aria-live` 通知

### 3.9 交互组件默认态声明规则

**任何具有交互触发行为的组件（如下拉菜单、弹窗、抽屉等），必须同时声明"默认态"与"触发后态"，且两者不可混写。**

| 规则 | 说明 |
|-----|------|
| 默认态 | 页面初始渲染时用户看到的内容。**只声明触发元素本身**（图标/按钮/文字），不包含触发后展开的内容 |
| 触发后态 | 用户操作（click/hover 等）后展示的内容（下拉项、弹窗内容等） |
| 禁止混写 | 不得将"触发后才展示的内容"描述为默认态布局的一部分 |

**常见错误示例（禁止）**：

> 语言切换区默认态：`文A 图标 + English + 中文`（将下拉选项写入默认态布局）

**正确示例**：

> 默认态：`文A 图标`（仅图标）
> 触发后态：下拉菜单，包含 `English` / `中文` 两个选项

### 3.10 组件度量完整性（强制）

#### 3.10.1 flex 容器 5 维度强制声明

**每个声明了 `display: flex` 的容器，必须同时声明以下 5 个维度**，缺少任何一项均视为不完整：

| 维度 | CSS 属性 | 说明 |
|-----|---------|------|
| ① 尺寸 | `width` / `height` / `align-self: stretch` 等 | 容器自身占据的空间 |
| ② 外间距 | `margin` / `gap`（来自父容器） | 与相邻元素的间距 |
| ③ 主轴对齐 | `justify-content` | 子元素沿主轴（flex-direction 方向）的排列方式 |
| ④ 交叉轴对齐 | `align-items` | 子元素沿交叉轴的对齐方式 |
| ⑤ 子元素布局 | `flex-direction` / `gap`（子元素间距）/ `flex-wrap` | 子元素的排列方向与间距 |

**降级**：若某维度在当前设计稿中明确不需要自定义（如继承父级），须显式标注"继承父级"或"不适用"，不可留空。

#### 3.10.2 等分布局对齐表达规则

当容器内子元素**等分排列**时（如 Tab 均分、列表列等宽），**禁止使用对称性语义描述对齐方式**（如"两端对齐""居中"），必须明确相对哪个子容器声明。

| 禁止写法 | 正确写法 |
|---------|---------|
| `justify-content: space-between`（无注释） | `justify-content: space-between`，注释：每个 Tab 子容器各自 `flex: 1`，文字在子容器内 `text-align: center` |
| "左右两端对齐" | "每个 Tab 独立容器，宽度等分（`flex: 1`），内容在容器内水平居中" |
| "均匀分布" | "4 列等宽，每列 `width: 25%`，内容 `justify-content: center`" |

**规则**：等分布局的对齐描述必须回答两个问题：
1. **父容器**如何分配空间给子容器？（`flex: 1` / `width: 25%` 等）
2. **子容器内部**内容如何对齐？（`text-align` / `justify-content` / `align-items`）

#### 3.10.3 ASCII 图限制说明

ASCII 图（如布局示意图）**只用于表达结构层级关系**，不用于表达间距数值、对齐方式或组件状态。

| ASCII 图可表达 | ASCII 图不可表达（须在间距表中另行声明） |
|-------------|----------------------------------|
| 容器嵌套层级 | 具体 padding / margin / gap 数值 |
| 组件位置关系（上下左右） | justify-content / align-items 取值 |
| 哪些元素并列 / 哪些元素嵌套 | 等分还是固定宽度 |

#### 3.10.4 含文字组件的字体属性内联规则

**§4.2 间距表中，凡是含有文字的组件行，必须在该组件的行组内直接内联关键字体属性**，不得仅在 §4.3 字体表中单独声明。

**原因**：agent 按组件逐行读取 §4.2 生成 CSS 时，不会自动回查 §4.3。若字体属性仅在 §4.3，则该属性极易被遗漏，导致继承浏览器或父级默认值（通常为 16px）。

**必须在 §4.2 内联的字体属性**：

| 属性 | 说明 |
|-----|------|
| `font-size` | **最高优先级**，必须显式设置，禁止继承 |
| `line-height` | 影响组件高度计算 |
| `font-weight` | 影响视觉粗细 |
| `color`（文字色） | placeholder / 正常态文字色需区分时必填 |

**内联格式示例**：

```markdown
| 输入框 | font-size    | 14px（⚠️ 必须显式设置，禁止继承父级或浏览器默认 16px） | §4.3 ✅ |
| 输入框 | line-height  | 22px | §4.3 ✅ |
| 输入框 | font-weight  | 400  | §4.3 ✅ |
| 输入框 placeholder | color | #B7C1D1（--text-color-placeholder） | §4.3 ✅ |
```

**降级**：若某组件字体完全继承父级（如卡片内所有文字统一由父级 `font-size` 控制），须在组件行注明"字体继承父级 [组件名]"，不可留空。

## 4. AC 覆盖反查(§11)

对 `requirement.md §8` 的每个 AC,在生成本节时回答:

> "这条 AC 在哪个页面 / 哪个状态 / 哪个交互中体现?"

| 情况 | 处理 |
|-----|------|
| AC 描述用户可见行为 | 必须能映射到具体页面+状态/交互 |
| AC 描述系统内部规则(如"密码加密存储") | 显式标注"无 UI 体现",在覆盖表中注明理由 |
| AC 描述边界值/极端情况 | 必须在对应页面的"极端数据"或"错误"状态中体现 |

输出表格:

```markdown
| AC ID | 对应页面/组件 | 对应状态/交互 | 覆盖? |
|------|------------|------------|------|
| AC-XXX-001 | §3.2 [组件名] | 表单提交 + 失败标黄 | ✅ |
| AC-XXX-007 | (后端规则,无 UI 体现) | - | N/A |
```

## 7. 不要做的事

- ❌ 在 ui-spec 重新定义数据字段类型(应引用 data-contract)
- ❌ 跳过"5 态"中的某一态(尤其是"极端数据"最常被跳过)
- ❌ 用"漂亮""现代""简洁"这种空形容词描述设计
- ❌ 假设角色权限(应严格按需求 §3 翻译)
- ❌ 给状态机的"已归档"配亮色(终态应低饱和)
- ❌ 跳过移动端章节(即使不支持也要明说)
- ❌ 在文案规范里用术语的同义词(漂移)
- ❌ **含文字的组件在 §4.2 间距表中不声明 `font-size`，仅在 §4.3 字体表中单独列出**
  - 反例：§4.2 输入框行只写 height/padding/border，font-size 只在 §4.3 出现
  - 后果：agent 读 §4.2 生成 CSS 时不回查 §4.3，font-size 未设置，继承浏览器默认 16px，placeholder 和输入文字偏大
  - 参见：§3.10.4（含文字组件字体属性内联规则）
  - 反例：`display: flex`（无主轴/交叉轴对齐声明）
  - 后果：agent 默认补全为 `justify-content: flex-start` + `align-items: stretch`，与设计稿实际对齐方式不符，导致实现偏差
  - 参见：§3.10.1（flex 容器 5 维度强制声明）
- ❌ **用对称性语义描述等分布局的对齐方式**（如"两端对齐""均匀分布"）而不说明相对哪个子容器
  - 反例："Tab 两端对齐" / `justify-content: space-between`（无注释）
  - 后果：agent 无法区分"父容器分配方式"与"子容器内对齐"，实现时在父级 space-between，子容器内容不居中，产生错位
  - 参见：§3.10.2（等分布局对齐表达规则）

## 8. Figma 文件结构规范

### 8.1 整体组织方式

采用**单 File + 多 Page** 方案：每个需求对应 Figma 文件中的一个 Page，所有需求共享同一个组件库与样式规范。

```
Figma Project: Smart Site
└── File: Smart Site - 设计稿
    ├── Page: _封面 & 变更日志        ← 记录每次设计变更
    ├── Page: _组件库 & 样式规范      ← 全局共享，以 _ 前缀标识
    ├── Page: REQ-001 [需求名称]
    ├── Page: REQ-002 [需求名称]
    ├── Page: REQ-003 [需求名称]
    └── ...
```

### 8.2 单个 Page 内部结构

每个需求 Page 内按以下 Section 组织：

```
REQ-XXX [需求名称]
├── Section: 流程图 & 状态机
├── Section: PC 端
│   ├── Frame: [页面名] - 默认态
│   ├── Frame: [页面名] - 空态
│   ├── Frame: [页面名] - 加载态
│   ├── Frame: [页面名] - 错误态
│   └── Frame: [页面名] - 极端数据态
├── Section: App 端（如有）
│   └── Frame: ...
├── Section: H5 端（如有）
│   └── Frame: ...
└── Section: 标注说明 & 交互备注
```

### 8.3 链接填写规范

在 `ui-spec.md §10 Figma 与原型链接` 中，填写 **Frame 级链接**（非 File 级），精确定位到对应画板：

- 在 Figma 中右键点击目标 Frame 或 Section → `Copy link`
- 粘贴到 `ui-spec.md`，格式示例：

| 平台 | 页面 | Figma 链接 |
|------|------|-----------|
| PC   | 登录页 | https://www.figma.com/file/xxx/Smart-Site?node-id=xxx |
| App  | 登录页 | https://www.figma.com/file/xxx/Smart-Site?node-id=xxx |

### 8.4 命名规则

| 对象 | 命名格式 | 示例 |
|------|---------|------|
| Page | `REQ-XXX 需求名称` | `REQ-001 登录注册` |
| Section | 平台名 或 功能分区 | `PC 端`、`流程图 & 状态机` |
| Frame | `页面名 - 状态` | `登录页 - 默认态` |
| 全局 Page | `_名称`（下划线前缀） | `_组件库 & 样式规范` |
