## DEC-001：工序任务采用三态不可逆工作流

**决策日期**：2026-03

**背景和问题**
施工现场的工序任务需要反映真实施工进度，状态一旦推进就不应允许回退，以保证进度数据的可信度和审计完整性。早期方案中考虑过允许 CM 将"进行中"退回"未开始"，但会引发进度虚报风险。

**可选方案**
1. **三态不可逆**：`not_started → in_progress → completed`，不允许逆转
2. **三态可逆**：允许 CM 将状态退回上一态
3. **多态**：增加"暂停""取消"等中间态

**最终决策**
采用方案 1（三态不可逆）。理由：施工进度具有现实不可逆性；允许退回会引发进度数据造假；中间态增加复杂度，当前阶段不必要。

**约定细节**
- 枚举值：`not_started` / `in_progress` / `completed`
- `completed` 为终态，后端拒绝任何状态逆转请求，返回 `422`
- APP 端 Start Task 无需确认弹框（低风险操作），Mark as Complete 必须有确认弹框（不可逆）
- 详见 `global-rules.md §14.4`

## DEC-002：任务进度基于工序权重加权计算

**决策日期**：2026-03

**背景和问题**
不同工序的工作量差异很大，简单按"已完成工序数 / 总工序数"计算进度会严重失真（如一个大型主体工序和一个小型扫尾工序权重相同）。

**可选方案**
1. **等权计数**：进度 = 已完成数 / 总数
2. **加权计算**：进度 = Σ已完成工序权重 / Σ所有工序权重
3. **人工填报**：PM/CM 手动输入进度百分比

**最终决策**
采用方案 2（加权计算）。理由：更真实反映工程量；权重由工序模板带出，减少人工干预；计算逻辑透明可审计。

**约定细节**
- 权重字段范围：1～10 整数
- 权重默认值从工序模板带出，CM 可手动修改
- 进度公式：`Σ(completed工序weight) / Σ(all工序weight) × 100%`
- 只统计 `status = completed` 的工序，`in_progress` 不计入
- 详见 `global-rules.md §14.5`

## DEC-003：已完成任务通过 is_locked 防止修改

**决策日期**：2026-03

**背景和问题**
工序任务进入 `completed` 状态后，其字段数据（计划时间、实际时间、权重等）应被冻结，防止事后篡改影响进度审计。

**可选方案**
1. **数据库级锁**：在 DB 层拒绝 UPDATE
2. **应用层 is_locked 标志**：后端读取 `is_locked = 1` 时拒绝修改，返回 403
3. **仅前端禁用**：UI 禁用编辑按钮，但后端不做校验

**最终决策**
采用方案 2（应用层 is_locked）。理由：数据库级锁运维成本高；仅前端禁用安全性不足；应用层标志灵活且可审计。

**约定细节**
- 状态变为 `completed` 时，后端同步将该记录 `is_locked = 1`
- 对 `is_locked = 1` 的记录发起任何修改操作，返回 `403`
- 前端收到 403 时展示"该记录已锁定，无法修改"提示
- 详见 `global-rules.md §14.4`

## DEC-004：多端采用独立设计语言，Token 不继承

**决策日期**：2026-03

**背景和问题**
PC 端使用 Element UI（桌面扁平风格），APP 端使用 UNIAPP（iOS/Android 圆润风格），两端面向不同用户、不同使用场景，强行共享一套 Token 会导致两端都不自然。

**可选方案**
1. **共享 Token，各端可覆盖**：定义基础 Token，各端在其上做差异化覆盖
2. **完全独立**：三端各自维护独立的 Token 文件，无继承关系
3. **只共享品牌色**：主品牌色共享，其余完全独立

**最终决策**
采用方案 2（完全独立）。理由：PC 和 APP 设计语言差异过大，共享层的价值低于维护成本；独立管理更清晰，Figma 文件也是独立的。

**约定细节**
- Token 文件：`design-tokens-pc.json` / `design-tokens-app.json` / `design-tokens-h5.json`
- 三端汇总对比文件 `design-tokens.json` 仅供参考，不作为开发依据
- 跨端一致性靠品牌层（同一套视觉语言原则），而非 Token 继承
- 详见 `outputs/ui/shared/tokens/design-tokens.json`

## DEC-005：物料出场库存不足直接阻断提交

**决策日期**：2026-04

**背景和问题**
物料出场登记时，如果出场数量超过当前库存，有两种处理方式：阻断提交，或允许提交但标记为"超出预警"。施工现场物料管控需要严格，超卖库存会导致账实不符。

**可选方案**
1. **硬阻断**：库存不足时前端实时校验 + 后端拒绝提交，返回 400
2. **软警告**：允许提交但标记为异常，事后人工处理
3. **审批流**：超出库存需要走审批流程

**最终决策**
采用方案 1（硬阻断）。理由：物料账实必须一致，软警告容易被忽视；审批流在当前阶段复杂度过高，留作后续迭代。

**约定细节**
- 前端提交前实时调用库存查询接口做预校验
- 后端提交时再次校验，库存不足返回 `400`，`message` 说明当前库存数量
- 物料调拨功能留作后续迭代，不在当前版本范围内
- 24 小时内可撤销出场登记，撤销后库存自动回补
- 详见 `requirements/pc/REQ-015-pc.md`

## DEC-006：仓库按职责分阶段分离，现阶段保持单仓

**决策日期**：2026-05-01

**背景和问题**
随着项目迭代，业务需求、设计体系、agent 规则框架三者职责不同、变更频率不同，长期放在一起会导致职责模糊和维护困难。

**最终决策**
现阶段保持单仓，待触发条件满足后按计划分离：

| 未来仓库 | 内容来源 | 触发时机 |
|---------|---------|---------|
| `smart-site-design` | `outputs/ui/shared/tokens/` + 组件规范 | 设计体系规则稳定后 |
| `ai-doc-framework` | `rules/ §1–§13` + `templates/` | 第二个项目启动时 |

**约定细节**
- 分离时只需修改各 `rules/*.md` 头部的 `继承:` 声明（约 7 处），业务文档零改动
- 详见 `global-rules.md §14.10`

## DEC-007：业务文档只引用语义名，禁止硬编码具体值

**决策日期**：2026-05-01

**背景和问题**
如果业务文档直接写入颜色值或尺寸值，未来 Token 变更时需要逐一修改所有业务文档，维护成本极高，也会阻碍仓库分离。

**最终决策**
业务文档只引用语义名，具体值集中在 Token 文件和 `global-rules.md §14` 中维护。

**约定细节**
- ✅ 正确：`statusTagType()` 返回 `"primary"`；颜色见 `global-rules §14.4`
- ❌ 错误：直接写 `color: #6892ff`
- 此规则对所有 agent 生成的文档强制生效
- 详见 `global-rules.md §14.10`
